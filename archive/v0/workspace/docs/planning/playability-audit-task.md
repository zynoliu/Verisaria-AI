# 测试任务：可玩性盘点 —— 一局真内容包玩下来，好不好玩？

> 给测试 Agent 的任务简报。**这次和以往所有跑都不同**：以前是验「某条机制链对不对、FALLBACK=0」；
> 这次是**探索式地玩，报告体验**。我们要的不是「机制对」的证据（那已经反复有了），而是
> **「一局真内容包玩下来连贯、可信、有分量、好玩」的证据**——或者，更可能的，**哪里不好玩**。

## 背景：为什么是这一跑

机制已经很多了：动态世界模型（涌现前置/ledger/护送/world-change）、活世界（变速时钟/天气/作息/沉浸注入）、
LLM 对白、关系、agenda/立场。**但「好玩」基本没验过**——动态世界模型只在 `escort_proving_ground` 这个
**为闭环而生的干净夹具**上盖过章；内容更丰富的 tidebreak/skyglass 当初是**卡住**的，被归为「内容层」搁置。

所以我们证明了「机制对」，没证明「玩起来好玩」。**这一跑就去拿那个证据（或反证）。**

## 怎么玩

- **选一个有血肉的包**：用你手上**内容丰富、有真实张力**的（`tidebreak_quarantine_harbor` /
  `skyglass_memory_inquest`，或新捏一个有质感的）。**别用 frostgate / escort_proving_ground**——它们太薄、太
  干净，验不出体验。
- **像一个好奇的玩家那样玩，不是执行脚本**：读开局、顺着 central_tension 走、找 NPC 搭话、试探世界、推一推、
  看会发生什么。**别为了闭某条链而优化**——就当你第一次玩这个世界，跟着兴趣走。
- **带点批判/对抗**：故意试边界、问刁钻问题、做意料外的事、回头找之前的 NPC 看它记不记得。**目的就是逼出
  「哪里出戏、哪里断头、哪里无聊」**——最有用的报告是「我在这儿会想退出」。
- **玩够一个像样的弧**：约 30–50 拍，或玩到「一个自然的段落收束 / 卡死 / 你失去兴趣」为止。真机 + `--log` +
  存 transcript。时间够可以跑两个包对照。

## 盯什么（七个体验维度，逐个给判断）

1. **连贯性**：世界自洽吗？NPC 的话 / world 变化，对照刚发生的事**说得通**吗？有没有失忆、自相矛盾、前后对不上？
2. **节奏**：拖不拖？有没有大段空拍 / 重复 / 无事发生？有意思的事是否以合适的密度发生？变速时钟让对话/快进的体感对吗？
3. **NPC 可信度**：NPC 像有立场/记忆/性格的人，还是会**跑题**（已知软肋）、重复、出戏、破人设？沉浸注入后它们接住
   时辰/天气了吗？
4. **选择有分量**：你做的事**看得见地起作用**吗（关系移动、世界变化、NPC 据此反应）？还是干啥都无所谓、世界无动于衷？
5. **出戏 / 断头 / 无聊**：具体哪些时刻你会**被拉出戏 / 撞墙没路走 / 觉得没意思**？贴原文。
6. **惊喜 / 涌现**：有没有**真正出乎意料、感觉世界活着**的瞬间（动态世界模型 + 活世界本该制造这些）？还是一切都
   平淡可预测？
7. **可上手**：开局你**看得懂处境、知道能干嘛、明白利害**吗？焦点/agenda/叙述把你领进去了吗？

## 报告请包含

- **⭐ 按「伤好玩」程度排序的摩擦清单**（最重要）：每条给 ——「发生了什么（贴原文）」+「属哪个维度」+
  「多伤：致命 / 明显 / 轻微」+（看得出的话）「疑似成因」。**这份清单就是我接下来动手的输入。**
- 七维度各一句总判（好 / 凑合 / 差 + 一句理由）。
- **一句话玩家感受**：作为玩家，这一局整体「好玩吗」？最想让它改的**一件事**是什么？
- 背景对错（次要，简记即可）：FALLBACK 计数、有无崩溃/死锁/断头、tick 超时数。
- 新 `*.log` + transcript + driver 放 `reports/<新目录>/`。

## 一句话目标

拿到**「一局真内容包到底好不好玩」的第一手证据**，以及**最伤体验的几处摩擦（带原文、排好序）**——好让开发据此
去修最该修的，而不是再加一个齿轮。**坦诚、批判，找「哪里会让人退出」，比找「哪里能跑通」有用得多。**

### 关注点（逐条回答）

1. **⭐ 怀疑能不能降**：当玩家**真诚/讲理/示善意**时，对象的 `suspicion` 是否出现过**下降**（负增量）、`trust`

2. **⭐ 旁观者是否不再爆表**：全程没搭话的乔巡长，怀疑是否**明显低于**你真正在交涉的对象（首跑它 0.77 全场最高）？

3. **选择是否有分量了**：现在「有效行动（真诚铺垫 + 降怀疑 + 建信任）」是否**真能把世界推动一点**——哪怕只是

4. **#1 证据线能否走通（引擎 + 内容一起看）**：若**光靠引擎修复**证据仍拿不到，**请你以内容作者身份调一调这个包**

   「引擎+内容一起，证据线现在能不能走通到 `⟳FLIP`」。**这一条明确允许你改 pack 内容**（注明改了什么）。
5. **回归抽查**：#2/#3/#4/#7 在真机是否确实好了（无凭空说话、按名移动通、无原始 id、开局有 drive）？`FALLBACK=0`？

### 报告请包含

- ⭐ 怀疑下降的对照例（真诚 → suspicion 负增量 / trust 升）：成立 / 仍只升不降。
- ⭐ 旁观者 vs 当事人 终局 suspicion 排名（旁观者是否不再最高）。
- 选择是否有分量了（主观 + 一个「世界被推动」的实例，若有）。
- #1：光引擎够不够；不够则你调了哪些 pack 内容、之后能否走通到 `⟳FLIP`。
- #2/#3/#4/#7 回归抽查 + `FALLBACK` 计数。
- 新 `*.log` + transcript + driver 放 `reports/<新目录>/`。

### 一句话目标

**「越使劲 NPC 越怀疑我、证据全锁、选择没分量」翻过来**；并判明把证据线推到真闭环还差不差一层**内容调参**。

## ⏱ 第三跑（commit 591b785 起）—— F1 收敛护栏：原始 fixture 不调内容能否自己闭合

二跑结论：证据线**「光引擎不够、差一层内容调参」**——真因是 arbiter 临场涌现 `pump_failure_disclosed_publicly`，
和作者 `pump_failure_disclosed` **互不相认**（满足一个不顶另一个），且不断加码（存档→公示→广播→联署…）。你当时靠
authored 副本手搓标签救活。

已上 **F1 两道确定性引擎闸**（`_register_dynamic_prerequisite`，commit `591b785`）：
- **去重复用**：新前置若与已有变量**语义近重复**（id-stem 包含，如 `pump_failure_disclosed` ⊂
  `pump_failure_disclosed_publicly`；或新关键词已命中已有 label）→ **复用已有变量**，把新措辞折进其 keywords。
- **封顶**：单个终态最多派生 `_MAX_PREREQS_PER_TERMINAL=2` 个前置，挡无限细分。

**这一跑就盯一件事：回到原始 `tidebreak_quarantine_harbor.json`（不是 authored 副本、不调任何标签/关键词），
F1 能不能让证据链自己闭合。**

> ⚠ F1 是确定性的，但**它能否命中真实 arbiter 行为取决于 LLM 怎么命名近重复**：若 LLM 把重复变量命名成
> 派生 id（`..._publicly`）或带重叠关键词，去重就生效；若它造一个**毫不相关的新 id**，id-stem 兜不住——这正是
> 真机要回答的。所以请**贴 arbiter 实际涌现的 var_id 原文**。

### 怎么跑

**原始 fixture**（仅可注入 world_premise 那几个无关开关，**不许动 world_state_vars 标签/关键词**），复刻二跑那条
endgame：把「三号泵闸事故」证据公开 → 去找林槐请求撤销征船令。driver 每拍打印 `/world`（全 world-var 值）+ 把每条
arbiter 涌现的 `new_prerequisite var_id` 落原文。

### 关注点（逐条回答）

1. **⭐ 去重是否命中**：arbiter 这次涌现的「公开」类前置 var_id **叫什么**（贴原文）？引擎是否把它**复用**进了已有
   `pump_failure_disclosed`（日志 `new_prerequisite proposed but NOT registered (dup/...)` 或 `/world` 里**没有**
   第二个公开类变量）？还是又造了个独立变量？
2. **⭐ 链能否自己闭合**：不调内容标签，`pump_failure_disclosed ⟳FLIP → tow_order_halted ⟳FLIP` 能否走通？还是
   仍卡在两变量互不相认（说明 LLM 命名躲过了 id-stem，需要更强语义去重）？
3. **封顶是否挡住加码**：林槐是否不再「存档→公示→广播→联署」一层层加新前置（同一终态动态前置数 ≤2）？贴
   `serves=tow_order_halted` 的动态前置个数。
4. **没误伤**：正常多前置场景（如证人作证 + 文件归档是两件事）是否仍各自成立、没被错误合并？
5. 回归：`FALLBACK=0`？无崩溃/死锁？

### 报告请包含

- ⭐ arbiter 涌现的「公开」var_id 原文 + 是否被复用进 `pump_failure_disclosed`（成立/躲过）。
- ⭐ 不调内容，证据链能否自己 `⟳FLIP` 到 `tow_order_halted`（成/卡 + 原因）。
- 封顶：单终态动态前置个数；加码是否停了。
- 有无误伤（合并了不该合并的）。
- `FALLBACK` 计数；新 `*.log` + transcript + driver 放 `reports/<新目录>/`。

### 一句话目标

确认 F1 在真机**让 arbiter 不再造近重复变量、不再无限加码**，从而**原始 fixture（零内容手搓）下证据链能自己闭合**；

## ⏱ 第四跑（commit 18e6674 起）—— 充分性闭环：原始 fixture 真能自己闭了吗

三跑结论：F1 的「id-stem 去重 + 数量封顶」**必要但不充分**——真病根是 arbiter 在 reason 里**软性**不断发明新前置
（id 不相关、躲过去重；数量 cap 反噬成「无路可走」死结 + 误伤独立前置）。已按用户拍板「**prompt 为主 + 结构兜底**」
重做（commit `18e6674`）：

- **回退数量封顶**（保留 id-stem 去重）。
- **prompt 硬规则 (d)**：每个终态把「**已满足的前置**」喂进裁定 prompt，规则——已满足前置之上**不得再加码、必须
  success**，否则即「软性出尔反尔」。
- **引擎兜底**：`serves` 记录每前置所属终态；当某终态**所有已声明前置全为真**，即便 arbiter 仍 partial 也
  **强制 success**（反作弊红线经用户同意放宽为「success 或 声明前置被真满足」——前置是真满足、非 bluff）。

**这一跑就盯一件事：回到原始 `tidebreak_quarantine_harbor.json`（不调任何 world_state_vars 标签/关键词），证据链
能不能自己闭到 `tow_order_halted ⟳FLIP`。**

### 怎么跑

原始 fixture（仅注入 world_premise 无关开关），复刻三跑那条 endgame：把三号泵闸事故证据公开 → 找林槐撤征船令。
driver 每拍打印 `/world` 全量 + 落每条 arbiter 涌现的 `new_prerequisite var_id`，并在日志里抓 `sufficiency backstop`
这行（引擎兜底触发标记）。**务必跑到终态或明确卡死再停，必出 report.md。**

### 关注点（逐条回答）

1. **⭐ 链能否自己闭**：不调内容，`pump_failure_disclosed ⟳FLIP → tow_order_halted ⟳FLIP` 能否走通？终局 `/world`
   两个终态是否都 True？（三跑此处两个都 False。）
2. **兜底 vs prompt 谁起的作用**：闭合是靠 **arbiter 自己据规则 (d) 判了 success**（日志无 `sufficiency backstop`），
   还是靠**引擎兜底强制**（日志有 `sufficiency backstop: <var> … forced success`）？两种都算闭，但要分清——若全靠
   兜底，说明 prompt (d) 在真机仍没 hold 住（可接受，兜底就是为此）。
3. **没误伤 / 没乱翻**：终态是否**只在其声明前置真满足后**才翻？有没有出现「前置没满足却被强制翻」（那是兜底逻辑 bug）
   或「把不相干的终态也翻了」？贴反例（若有）。
4. **加码是否止住**：森工/林槐在前置满足后是否**不再发明新条件**（reason 里不再「还要再走一道…」）？还是仍嘴上加码、
   但被兜底无视掉了？
5. 回归：`FALLBACK=0`？无崩溃/死锁？

### 报告请包含

- ⭐ 不调内容，证据链能否自己 `⟳FLIP` 到 `tow_order_halted`（成/卡 + 终局 `/world`）。
- 闭合是 prompt (d) 还是引擎兜底起效（贴 `sufficiency backstop` 日志有无）。
- 有无误伤/乱翻（终态只在声明前置真满足后翻）。
- 加码是否止住。
- `FALLBACK` 计数；新 `*.log` + transcript + driver 放 `reports/<新目录>/`。

### 一句话目标

确认「prompt (d) + 引擎兜底」让**原始 fixture（零内容手搓）下港口证据链自己闭到 `tow_order_halted ⟳FLIP`**，且终态
**只在声明前置真满足后**翻（反作弊精神守住）；并分清闭合主要靠 prompt 还是靠引擎兜底，好判断 prompt (d) 在真机够不够硬。

## ⏱ 第六跑（commit 50a356c 起）—— 被施压的当事人，真诚现在撬得动吗（#2/F2 覆盖）

五跑暴露：#5 的「真诚降怀疑/升信任」在**旁观者**成立，但在**被追问/调查的当事人**身上不触发——苏拉
suspicion 死钉 0.2、trust 0.000，连「我不想毁你的兰房」这种明确给台阶也纹丝不动。appraiser **看得到**玩家说的
内容（事件 summary 含 content），所以是 LLM 判断盲点：把「这人在查我」压过了「TA 给了台阶」。

已强化 appraisal prompt（commit `50a356c`）：明确把**给台阶/承诺不损害你利益/卸下威胁**列为降怀疑·升信任的触发，
并点明**即便对方正在调查你**、只要真展现善意也应松动，判「毫无变化」是不真实的。**这是 LLM 行为改动，FakeLLM 验不出。**

**这一跑盯两件事，且要分清是「LLM 现在肯给增量了」还是「增量被关系存储的衰减吃掉了」：**

### 怎么跑

复刻五跑那条线（skyglass 谷地/兰房，或任一有「被施压的当事人」的线）。对一个起手就戒备的当事人（如苏拉），
**连着用真诚 / 给台阶 / 承诺不伤害 TA 的话谈几轮**。driver 务必开 `--log` 并抓 appraisal 日志行
（`<npc> appraises player: Δ{...} → {...}`，logger `verisaria...`，见 `_apply_appraisal_results`）——它**同时**给出
**LLM 原始增量 Δ** 和**应用后的关系值**，正好用来分辨两种情况。

### 关注点（逐条回答）

1. **⭐ 当事人现在动不动**：对被追问的当事人用真诚/给台阶，appraisal 是否出现 **负 suspicion / 正 trust 增量**？
   贴 2–3 条 `Δ{...}` 原文（对照五跑：苏拉全程 Δ 为空 / 0）。
2. **⭐ 分清 LLM vs 衰减**：若 Δ 里**有**负 suspicion（如 -0.05），但 `→` 后的 suspicion **几乎没降**（0.2→0.19），
   那是**关系存储的下行衰减**（`delta*cur`，低值处极慢）吃掉了——**贴「Δ 多大 / 实际降了多少」的对照**。这决定
   下一步是否要调衰减公式（我没动它，怕扰动 #5）。若 Δ 本身仍是空/0，那是 prompt 还没 hold 住。
3. **F2 覆盖面**：一轮里**有质量的说服 tick**（真诚、对口、给台阶）中，**多大比例**真的让关系动了「看得见」的量？
   （五跑是「很多仍 +0.000」。）
4. **回归**：#5 没被带坏——旁观者真诚仍降怀疑、施压/威胁仍升怀疑、没有「一句好话清零戒备」的失衡？贴一两个反例（若有）。

### 报告请包含

- ⭐ 当事人 Δ 对照（负 suspicion/正 trust 是否出现）：成立 / 仍空。
- ⭐ LLM-给了-Δ-但-衰减吃掉 的对照（Δ vs 实际位移），判是否要调衰减。
- F2 覆盖面比例（粗估即可）。
- #5 回归抽查。
- 新 `*.log` + transcript + driver 放 `reports/<新目录>/`。

### 一句话目标

确认强化后的 appraisal prompt 让**被施压的当事人也会对真诚/给台阶松动**（负 suspicion/正 trust 真出现），并**分清**
剩下的「不够动」是 prompt 没 hold 还是衰减公式太黏——好决定下一步调 prompt 还是调衰减。

## ⏱ 第七跑（commit 1d71434 起）—— 杠杆模型：有筹码才撬得动当事人

六跑是干净的否定 + 精确诊断：**不是衰减**（familiarity 全额落地），**是 LLM 先验**——硬顶 prompt 让被调查的
当事人对纯软话松动反而 backfire（苏拉 给台阶 → suspicion 0.2→0.292）。用户拍板**改模型**：有鬼的当事人对调查者
的软话保持戒备是真实的，**分量应来自筹码、不是嘴皮子**。已上**杠杆模型**（commit `1d71434`）：

- appraiser 现在收到一段「**对方握有的、与你利害相关的实据/筹码**」= 玩家在该 NPC 所管 var 上**已确立的 ledger
  事实**（`_player_leverage_over`）。
- prompt 分流：一般真诚仍走降怀疑通道；但当 NPC **戒备/被调查、且对方想从 TA 这儿得到什么**时——**纯空口软话被
  打折扣**（既不升也不降怀疑，修了六跑的 backfire），**唯有有上面「实据/筹码」撑腰的让步**才让戒备松动、生信任。

**这一跑就盯一件事，且必须分两种打法对照：**

### 怎么跑

同一条「被施压的当事人」线（skyglass 苏拉，或同类）。**两段对照**：
- **A 纯软话（无筹码）**：开局直接对当事人真诚/给台阶几轮——**此时玩家手里没有任何已确立事实**。
- **B 有筹码**：先去**把一条相关事实坐实**（让某个该 NPC 所管 var 进 ledger——即先做出一次 partial_success/取证，
  使 `_player_leverage_over` 非空），**再**对当事人用同样的真诚/让步。

driver 开 `--log` 抓 appraisal 行（`<npc> appraises player: Δ{...} → {...}`），并在每段标注「此刻玩家是否有筹码」。

### 关注点（逐条回答）

1. **⭐ A 段（无筹码）**：纯软话现在是否**不再让 suspicion 上升**（修好 backfire）、大体持平？贴 Δ（对照六跑的
   +0.05/+0.02）。当事人对空口软话保持戒备 = 预期正确行为，不是 bug。
2. **⭐ B 段（有筹码）**：玩家手里有了已确立事实后，同样的真诚/让步是否**真的让 suspicion 下降 / trust 上升**了？
   贴 A vs B 的 Δ 对照——**这是杠杆模型成不成的核心判据**。
3. **筹码信号是否喂到**：确认 B 段 appraiser 真收到了非空「实据/筹码」块（可在 prompt dump 或行为差异上印证）。
4. **回归**：#5 旁观者/一般真诚仍降怀疑；施压/威胁仍升怀疑；没有「有筹码就一句话清零戒备」的过冲。
5. `FALLBACK=0`？

### 报告请包含

- ⭐ A（无筹码）Δ：软话是否已不再加深怀疑（backfire 修没修）。
- ⭐ B（有筹码）Δ：真诚让步是否真撬动了 suspicion↓/trust↑（A vs B 对照）。
- 筹码信号确认到位。
- #5 回归抽查 + `FALLBACK` 计数。
- 新 `*.log` + transcript + driver 放 `reports/<新目录>/`。

### 一句话目标

确认杠杆模型成立：**没筹码时，软话不再 backfire、当事人合理地保持戒备**；**有筹码（已确立事实）时，同样的真诚让步
真能撬动当事人**（suspicion↓/trust↑）——让「选择有分量」来自取证而非嘴皮子；并确认 #5 没被带坏。
