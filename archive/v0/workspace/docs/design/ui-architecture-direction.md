# UI 架构方向（设计参考）

> 记录于 2026-06-01。这是 UI 专项**开工前**的方向性结论与约束，不是实现方案。
> 正式做 UI 时以本文为起点，再细化数据契约与接口定义。

---

## 0. 背景

`ui_prototypes/` 下已有 4 张设计图：TUI 原型、像素风 GUI 原型，以及各自的 storyboard。
评估结论：**质量高，且少见地"懂自己的引擎"**——几乎每个面板都对应一个真实存在的后端子系统，
storyboard 演的就是 frostgate 难民剧本。问题不在审美或系统理解，而在**架构落地的取舍**。

---

## 1. UI 元素 ↔ 引擎系统映射（已核对代码）

| UI 元素 | 对应引擎系统（真实存在） |
|---|---|
| 关系维度进度条（信任/怀疑/恐惧…） | Channel A `RelationshipStore`（6 维）；现有 `/relationship` |
| 记忆(近期) 带 Tick | `MemoryStore`（salience / recency） |
| 可见真相(某NPC所知) + 锁住条目 | world-book 派系分层真相 `WorldBookFilter`（**设计核心支柱**：信息不对称） |
| 议程 | Channel B agenda；现有 `/agenda` |
| 压力 % | campaign driver 压力 |
| 世界变化 / "边境关闭"牌子 | Channel C `world_vars`（`refugees_admitted`）；现有 `/world` |
| 行动/说服/威逼/交易/探索 | action 类型路由（部分 → arbiter） |
| 霜门哨站 / 队长布兰 / 温度 / 派系数 | frostgate pack + world state |

**含义**：TUI 需要的数据，引擎**绝大多数已经暴露成 `/命令`**。TUI ≈ 把这些命令面板化。

---

## 2. 架构方向：无头引擎 + 可插拔前端（认可）

用户倾向把 UI 做成**插件/适配器**，前端经接口兼容后端，方便切换 TUI/GUI、集成第三方 UI。
方向正确（headless core + adapter frontends），值得走。

**但要看清：插件机制是简单的一半，真正的活是「数据契约」。**

---

## 3. 真正的难点：结构化视图模型（数据契约）

当前边界是**"字符串化、表现耦合"**的：

- `_show_relationships()` / `_show_agenda()` / `_show_world()` 返回**格式化好的中文字符串**；
- `run_tick()` 返回 ResponseGenerator 拼好的**成品叙事串**。

插件 UI 要的是**结构化数据**（关系=6 个数值、议程=对象列表、叙事=事件数组），不是预渲染字符串。

> **核心工作**：把引擎产出抽成一层 **结构化、可 JSON 序列化、版本化的视图模型（DTO）**。
> 现有 `_show_*` 字符串方法应**降级为 TUI 自己的渲染器**，而不再是引擎职责。

"定义 TUI 需要什么" ≡ "定义这个接口"——两件事是同一件事。

---

## 4. 接口必须遵守的三条硬约束

1. **接口 = A5 授权边界（最重要）**
   接口只能返回**玩家合法可见的投影**。`可见真相(某NPC所知)` 面板要么是
   **调试/上帝视角**（正式对局关闭），要么数据来源改成**"玩家对该 NPC 的推测"**
   （来自玩家 belief，而非 `WorldBookFilter.filter_for_entity(该NPC)`）。
   **绝不能让漂亮面板把 NPC 私密知识漏给前端。** 把信息不对称的守卫放在接口层，一劳永逸。

2. **事件流，而非请求/响应**
   一拍 20~190s（PLAY-1），同步 `submit() -> 结果` 会冻死 UI。
   建议 `submit()` 立即返回 + UI **订阅事件流**（"思考中… / voss 开口 / 部分叙事 / 最终状态"）。
   既喂得动 GUI 动画，也天然适配第三方客户端。

3. **JSON 可序列化**
   视图模型设计成 JSON 可序列化 → "进程内 Python 接口（TUI/GUI）"与
   "跨进程/跨语言协议（第三方）"只是**换传输层**，不是重新设计。先进程内，留好后路。

---

## 5. 避免过早抽象（YAGNI / rule of three）

只有 TUI 一个消费者时就设计"通用插件接口"，容易抽错。建议：

> 先用 **TUI 一个真实消费者**把结构化 facade 跑通，接口"**由实现反推**"。
> 等像素 GUI 来了，从两者**共同需要**的东西里硬化稳定接口，再对外开放给第三方。

---

## 6. 排序

| 阶段 | 内容 | 理由 |
|---|---|---|
| **先做** | **TUI** | 数据≈现有命令面板化；对 PLAY-1 的慢宽容；最快验证"晒系统"这套 UX；定下数据契约 v0 |
| **后做** | 像素 GUI | 当上线门面；照搬 TUI 的数据契约；等玩法被验证值得再投美术 |

**像素 GUI 的额外成本（开工前必须认清）**：
- **缺坐标层**：引擎只有 `location_id`/`zone_id`，**无 x/y 像素位置**。俯视瓦片世界需新增
  "地点→瓦片布局/坐标"映射（pack 声明，或独立 layout 层）+ 立绘/精灵资源。
  **这是引擎之外另起的、不小的前端+美术工程。**
- 实时感会和 PLAY-1 的慢打架：华丽 GUI 每回合冻结两分钟，体感像卡死。不管哪个 UI，
  都得先设计好"思考中…"的等待反馈。

---

## 7. 开工前要先定死的开放问题

1. **`可见真相` 面板归谁的视角？**（调试视角 vs 玩家推测）——决定数据来源与 A5 实现，必须先选。
2. **走云端 API（计费）还是自有 GPU（省钱）**——影响 UI 的延迟假设与"等待反馈"设计（PLAY-1）。
3. **数据契约 v0**：列清每个面板要引擎给什么——现成的（关系/记忆/真相/议程/世界/压力均已暴露）
   vs 要新加的（坐标层、视角标记、事件流）。这是 UI 专项的第一块基石。
